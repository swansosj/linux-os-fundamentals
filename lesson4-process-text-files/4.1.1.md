## The `cut` Command — Extract Sections from Lines

`cut` extracts specific columns or fields from each line of a file.

### Basic Syntax

```
cut [OPTIONS] [FILE]
```

### Cut by Character Position (`-c`)

```bash
# Extract characters 1 through 10 from each line
cut -c 1-10 employees.csv

# Extract only the first character of each line
cut -c 1 employees.csv

# Extract characters 1-5 and 10-15
cut -c 1-5,10-15 employees.csv

# Extract from character 20 to end of line
cut -c 20- employees.csv

# Extract up to character 8
cut -c -8 employees.csv
```

### Cut by Field / Delimiter (`-f` and `-d`)

This is the most common usage — splitting lines by a delimiter and extracting fields.

```bash
# Extract field 1 (default delimiter is TAB)
cut -f 1 inventory.txt

# Extract field 1 using comma as delimiter
cut -d ',' -f 1 employees.csv

# Extract fields 1 and 3 (name and department)
cut -d ',' -f 1,3 employees.csv

# Extract fields 1 through 4
cut -d ',' -f 1-4 employees.csv

# Extract field 3 onwards (field 3 to end)
cut -d ',' -f 3- employees.csv

# Use colon as delimiter (e.g., /etc/passwd)
cut -d ':' -f 1 /etc/passwd            # Usernames only
cut -d ':' -f 1,3 /etc/passwd          # Username and UID
cut -d ':' -f 1,6 /etc/passwd          # Username and home directory
cut -d ':' -f 1,7 /etc/passwd          # Username and default shell
```

### The `--complement` Option

```bash
# Show everything EXCEPT field 2
cut -d ',' -f 2 --complement employees.csv

# Show everything EXCEPT fields 4 and 5
cut -d ',' -f 4,5 --complement employees.csv
```

### The `--output-delimiter` Option

```bash
# Change the output delimiter (input is comma, output is tab)
cut -d ',' -f 1,3,4 --output-delimiter=$'\t' employees.csv

# Change comma delimiter to pipe
cut -d ',' -f 1,2,3 --output-delimiter='|' employees.csv
```

### Practical Examples

```bash
# Extract all usernames from /etc/passwd
cut -d ':' -f 1 /etc/passwd

# Get IP addresses from a log file (assuming space-delimited, IP in field 1)
cut -d ' ' -f 1 server.log

# Extract dates from log entries
cut -d ' ' -f 1,2 server.log

# Combine cut with other commands
cat employees.csv | cut -d ',' -f 1,4 | sort
```

> **Teaching Note:** `cut` is column-oriented — it works on every line identically. Key points:
> - `-d` sets the delimiter (default is TAB, not space)
> - `-f` selects fields (columns after splitting by delimiter)
> - `-c` selects character positions
> - Fields and characters are **1-indexed**
> - `cut` cannot reorder fields — `cut -f 3,1` outputs fields in the order they appear (1, then 3)

## The `head` Command — Display the Beginning of a File

### Basic Usage

```bash
# Show first 10 lines (default)
head employees.csv

# Show first N lines
head -n 5 employees.csv
head -5 employees.csv            # Shorthand

# Show first N bytes
head -c 100 employees.csv       # First 100 bytes
head -c 1K employees.csv        # First 1 kilobyte

# Show all EXCEPT the last N lines
head -n -5 employees.csv        # Everything except last 5 lines
```

### Multiple Files

```bash
# Show first 5 lines of multiple files (adds headers automatically)
head -n 5 employees.csv inventory.txt server.log

# Suppress the filename headers
head -q -n 5 employees.csv inventory.txt

# Always show headers (even for single file)
head -v -n 3 employees.csv
```

### Practical Examples

```bash
# Quick preview of a large file
head -20 /var/log/syslog

# Get the header row of a CSV file
head -n 1 employees.csv

# Get header + first 5 data rows
head -n 6 employees.csv

# Pipe usage — show first 10 results of another command
ls -lt | head

# Show first 3 largest files
ls -lS | head -n 4     # includes the "total" line
```

> **Teaching Note:** `head` is great for quickly previewing file contents without loading the entire file. Pair it with `tail` to show different parts of a file. The `-n -N` syntax (negative number) is very useful — it means "everything except the last N lines."

## The `grep` Command — Search Text Using Patterns

`grep` (Global Regular Expression Print) searches for lines matching a pattern and prints them.

### Basic Syntax

```
grep [OPTIONS] PATTERN [FILE...]
```

### Basic Usage

```bash
# Search for a string in a file
grep "Engineering" employees.csv

# Search is case-sensitive by default
grep "engineering" employees.csv        # May return nothing
grep "Engineering" employees.csv        # Matches

# Case-insensitive search
grep -i "engineering" employees.csv

# Search multiple files
grep "error" server.log errors.log

# Search all files in a directory
grep "TODO" *.txt

# Search recursively through directories
grep -r "config" /etc/
grep -rn "config" /etc/                 # -n adds line numbers
```

### Common Options

```bash
# -n  Show line numbers with output
grep -n "ERROR" server.log

# -c  Count matching lines (don't show them)
grep -c "ERROR" server.log

# -l  Show only filenames that contain matches
grep -l "error" *.log

# -L  Show only filenames that do NOT contain matches
grep -L "error" *.log

# -v  Invert match — show lines that do NOT match
grep -v "comment" server.log
grep -v "^#" /etc/fstab             # Show lines not starting with #

# -w  Match whole words only
grep -w "the" words.txt             # Won't match "there" or "other"

# -x  Match entire lines only
grep -x "hello" words.txt

# -o  Show only the matching part (not the whole line)
grep -o "[0-9]\+" server.log

# -A N  Show N lines AFTER each match (context)
grep -A 3 "ERROR" server.log

# -B N  Show N lines BEFORE each match
grep -B 2 "CRITICAL" server.log

# -C N  Show N lines BEFORE and AFTER (context)
grep -C 2 "WARN" server.log

# --color  Highlight matches (usually default on Debian)
grep --color=auto "error" server.log
```

### Regular Expression Patterns

```bash
# ^  Match beginning of line
grep "^2026" server.log                 # Lines starting with "2026"

# $  Match end of line
grep "\.conf$" words.txt                # Lines ending with ".conf"

# .  Match any single character
grep "h.t" words.txt                    # Matches "hat", "hit", "hot"

# *  Match zero or more of the previous character
grep "go*d" words.txt                   # Matches "gd", "god", "good", "goood"

# []  Match any character in the brackets
grep "[aeiou]" words.txt                # Lines containing a vowel
grep "^[A-Z]" words.txt                 # Lines starting with uppercase

# [^]  Match any character NOT in the brackets
grep "[^0-9]" server.log               # Lines with non-numeric characters

# \  Escape special characters
grep "\." employees.csv                 # Match a literal dot
```

### Practical Examples

```bash
# Find failed login attempts
grep "Failed" /var/log/auth.log

# Search for an IP address
grep "192\.168\.1\." server.log

# Find lines with numbers
grep "[0-9]" employees.csv

# Find blank lines
grep "^$" employees.csv

# Count non-blank lines
grep -vc "^$" employees.csv

# Find lines between 2 patterns (show section)
grep -A 100 "START" server.log | grep -B 100 "END"

# Search command output (pipe)
ps aux | grep "apache"
dmesg | grep -i "usb"
history | grep "apt"
```

> **Teaching Note:** `grep` is arguably the single most-used text processing command in Linux. Students should master:
> 1. Basic string searching
> 2. `-i` for case-insensitive
> 3. `-v` for inverse
> 4. `-n` for line numbers
> 5. `-r` for recursive
> 6. `-c` for counting
> 7. Basic regex: `^` (start), `$` (end), `.` (any char), `[]` (character class)

## The `egrep` Command — Extended grep

`egrep` is equivalent to `grep -E`. It uses **Extended Regular Expressions (ERE)**, which means special characters like `+`, `?`, `|`, `(`, `)` work without escaping.

### Differences from Basic `grep`

| Feature          | `grep` (BRE)       | `egrep` / `grep -E` (ERE)  |
|------------------|---------------------|-----------------------------|
| One or more      | `\+`                | `+`                         |
| Zero or one      | `\?`                | `?`                         |
| Alternation (OR) | `\|`                | `\|` → `|`                  |
| Grouping         | `\( \)`             | `( )`                       |

### Basic Usage

```bash
# Alternation — search for multiple patterns (OR)
egrep "ERROR|WARN|CRITICAL" server.log
# Equivalent to: grep -E "ERROR|WARN|CRITICAL" server.log

# One or more (+)
egrep "[0-9]+" server.log               # One or more digits

# Zero or one (?)
egrep "colou?r" words.txt               # Matches "color" and "colour"

# Grouping with alternation
egrep "(Engineering|Marketing)" employees.csv

# Repeat N times {n}
egrep "[0-9]{3}" server.log             # Exactly 3 digits in a row

# Repeat range {n,m}
egrep "[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}" server.log
# Matches IP address patterns
```

### Practical Examples

```bash
# Find lines with email-like patterns
egrep "[a-zA-Z0-9.]+@[a-zA-Z0-9.]+" employees.csv

# Find multiple error levels at once
egrep -c "ERROR|WARN" server.log

# Find lines starting with specific words
egrep "^(GET|POST|PUT|DELETE)" server.log

# Find phone number patterns
egrep "[0-9]{3}-[0-9]{3}-[0-9]{4}" employees.csv

# Match words that repeat
egrep "(the|a) \1" words.txt

# Find dates in YYYY-MM-DD format
egrep "[0-9]{4}-[0-9]{2}-[0-9]{2}" server.log
```

> **Teaching Note:** Modern best practice is to use `grep -E` instead of `egrep` (which is deprecated on some systems). The key advantage of ERE is the `|` (alternation/OR) operator — searching for multiple patterns at once is extremely useful in log analysis. For this course, emphasize `grep -E "pattern1|pattern2"` as the go-to for multi-pattern searches.

## The `sort` Command — Sort Lines of Text

### Basic Usage

```bash
# Sort alphabetically (default)
sort words.txt

# Sort a CSV file
sort employees.csv

# Sort in reverse order
sort -r words.txt
```

### Numeric Sorting

```bash
# Numeric sort (treats content as numbers, not strings)
sort -n words.txt

# Without -n: "9" comes after "10" (string sort: "1" < "9")
# With    -n: 9 comes before 10 (numeric sort)

# Reverse numeric sort
sort -rn words.txt

# Sort by human-readable sizes (1K, 2M, 3G)
sort -h filesizes.txt
```

### Sorting by Field / Column

```bash
# Sort by field 2 (default delimiter is whitespace)
sort -k 2 words.txt

# Sort by field 4 (salary) numerically, comma-delimited
sort -t ',' -k 4 -n employees.csv

# Sort by field 3 (department) alphabetically
sort -t ',' -k 3 employees.csv

# Sort by field 4 numerically, then by field 1 alphabetically
sort -t ',' -k 4,4n -k 1,1 employees.csv

# Sort by specific character positions within a field
sort -t ',' -k 3.1,3.5 employees.csv
```

### Removing Duplicates

```bash
# Sort and remove duplicate lines
sort -u words.txt

# This is equivalent to:
sort words.txt | uniq
```

### Checking If Already Sorted

```bash
# Check if a file is already sorted
sort -c words.txt
# Outputs nothing if sorted, error message if not

# Check with no output — use exit code
sort -c words.txt && echo "Sorted" || echo "Not sorted"
```

### Stability and Advanced Options

```bash
# Stable sort (preserves original order of equal elements)
sort -s -t ',' -k 3 employees.csv

# Ignore leading blanks
sort -b employees.csv

# Case-insensitive sort (fold lowercase to uppercase)
sort -f words.txt

# Sort by month name (Jan, Feb, Mar...)
sort -M months.txt
```

### Practical Examples

```bash
# Sort /etc/passwd by UID (field 3)
sort -t ':' -k 3 -n /etc/passwd

# Find the 5 largest files in current directory
ls -lS | head -6

# Sort CSV by salary descending, show top 5 earners
sort -t ',' -k 4 -rn employees.csv | head -5

# Sort and deduplicate IP addresses from a log
cut -d ' ' -f 1 server.log | sort -u

# Sort by multiple columns: department, then salary descending
sort -t ',' -k 3,3 -k 4,4rn employees.csv

# Count unique entries after sorting
sort words.txt | uniq -c | sort -rn
```

> **Teaching Note:** Key points for `sort`:
> - Default is alphabetical/lexicographic (so "10" < "9" — use `-n` for numbers!)
> - `-t` sets the field delimiter, `-k` selects the sort key (field)
> - `-r` reverses, `-n` for numeric, `-u` for unique
> - `-k 3,3` means "sort by field 3 only" — without the second `3`, sort uses field 3 through end of line
> - `sort | uniq` is a classic pipeline pattern

## The `wc` Command — Word, Line, and Character Count

### Basic Usage

```bash
# Show line count, word count, and byte count
wc employees.csv
#  25  25 1024 employees.csv
# (lines) (words) (bytes) (filename)

# Count only lines
wc -l employees.csv

# Count only words
wc -w employees.csv

# Count only bytes
wc -c employees.csv

# Count only characters (different from bytes for multi-byte encodings)
wc -m employees.csv
```

### Multiple Files

```bash
# Count for multiple files (includes a total)
wc employees.csv server.log words.txt
#   25   25  1024 employees.csv
#  100  800  8192 server.log
#   50   50   400 words.txt
#  175  875  9616 total

# Count only lines for multiple files
wc -l employees.csv server.log words.txt
```

### Practical Examples

```bash
# Count how many files are in a directory
ls -1 | wc -l

# Count how many processes are running
ps aux | wc -l

# Count how many lines match a grep pattern
grep "ERROR" server.log | wc -l
# (This is equivalent to grep -c "ERROR" server.log)

# Count unique words in a file
cat words.txt | sort -u | wc -l

# Count how many users are on the system
cat /etc/passwd | wc -l

# Count how many lines do NOT contain "comment"
grep -v "comment" server.log | wc -l

# Find the longest line in a file
wc -L employees.csv         # -L shows maximum line length
```

> **Teaching Note:** `wc` is simple but extremely useful, especially in pipelines. Students commonly use `| wc -l` to count the output of other commands. Emphasize:
> - `-l` for lines (the most frequently used)
> - `-w` for words
> - `-c` for bytes
> - Pipe syntax: `command | wc -l` to count results

## Understanding Standard Streams

Every Linux process has three standard I/O streams:

| Stream            | File Descriptor | Default Destination | Purpose              |
|-------------------|:--------------:|---------------------|----------------------|
| Standard Input    | `0`            | Keyboard            | Input to the program |
| Standard Output   | `1`            | Terminal (screen)    | Normal output        |
| Standard Error    | `2`            | Terminal (screen)    | Error messages       |

```
              ┌──────────────────────┐
  stdin (0)   │                      │  stdout (1)
 ──────────>  │      COMMAND         │  ──────────>  Terminal
  (keyboard)  │                      │  
              │                      │  stderr (2)
              └──────────────────────┘  ──────────>  Terminal
```

> **Teaching Note:** Both stdout and stderr go to the terminal by default, which is why they look the same. Redirection lets us **separate them** and send each somewhere different. This is the foundation of all redirection.

## The `echo` Command and Basic Output

```bash
# Print text to stdout
echo "Hello, World!"

echo "My home directory is $HOME"
echo "Today is $(date)"
```

## Output Redirection

### `>` — Redirect stdout (Overwrite)

```bash
# Redirect output to a file (creates or OVERWRITES)
echo "Hello World" > greeting.txt

# Save command output to a file
ls -la > directory_listing.txt
date > current_date.txt

# Redirect output of a filter
grep "ERROR" server.log > errors_only.txt
sort employees.csv > sorted_employees.csv
```

> **⚠️ WARNING:** `>` **destroys** the existing content of the file. This is silent and unrecoverable.

### `>>` — Redirect stdout (Append)

```bash
# Append output to a file (creates if doesn't exist; preserves existing content)
echo "First entry" > logfile.txt
echo "Second entry" >> logfile.txt
echo "Third entry" >> logfile.txt

cat logfile.txt
# First entry
# Second entry
# Third entry

# Append command output
date >> activity.log
whoami >> activity.log
pwd >> activity.log

# Append filtered results
grep "WARN" server.log >> warnings_collection.txt
```

### `1>` — Redirect stdout Explicitly

`1>` is identical to `>` — the `1` explicitly refers to file descriptor 1 (stdout).

```bash
# These two are equivalent:
echo "Hello" > output.txt
echo "Hello" 1> output.txt

# Useful when you need to be explicit about WHICH stream you're redirecting
ls /etc /nonexistent 1> stdout.txt
# stdout goes to file, stderr still goes to terminal
```

### `2>` — Redirect stderr (Overwrite)

```bash
# Redirect only error messages to a file
ls /nonexistent 2> errors.txt
cat errors.txt
# ls: cannot access '/nonexistent': No such file or directory

# Suppress error messages by sending to /dev/null
find / -name "*.conf" 2> /dev/null
# Shows results without "Permission denied" errors cluttering output

# Separate stdout and stderr to different files
ls /etc /nonexistent 1> found.txt 2> notfound.txt
```

### `2>>` — Redirect stderr (Append)

```bash
# Append error messages to a log file
ls /fake_dir1 2>> error_log.txt
ls /fake_dir2 2>> error_log.txt
ls /fake_dir3 2>> error_log.txt

cat error_log.txt
# ls: cannot access '/fake_dir1': No such file or directory
# ls: cannot access '/fake_dir2': No such file or directory
# ls: cannot access '/fake_dir3': No such file or directory
```

### `&>` — Redirect Both stdout AND stderr (Overwrite)

```bash
# Send ALL output (stdout + stderr) to one file
ls /etc /nonexistent &> all_output.txt

# Equivalent older syntax:
ls /etc /nonexistent > all_output.txt 2>&1

# Suppress ALL output completely
command_that_might_fail &> /dev/null
```

### `&>>` — Redirect Both stdout AND stderr (Append)

```bash
# Append ALL output to a log file
echo "=== Run 1: $(date) ===" >> full_log.txt
ls /etc /nonexistent &>> full_log.txt

echo "=== Run 2: $(date) ===" >> full_log.txt
find / -name "shadow" &>> full_log.txt
```

### `2>&1` — Redirect stderr to stdout

This merges stderr INTO stdout so they go to the same place.

```bash
# Merge stderr into stdout, then redirect both to a file
ls /etc /nonexistent > combined.txt 2>&1

# THIS IS EQUIVALENT TO:
ls /etc /nonexistent &> combined.txt

# ORDER MATTERS:
# CORRECT — redirect stdout first, then merge stderr into it:
ls /etc /nonexistent > output.txt 2>&1

# WRONG — this sends stderr to where stdout WAS (terminal), then redirects stdout:
ls /etc /nonexistent 2>&1 > output.txt
# ^^^ stderr still goes to terminal!
```

> **Teaching Note:** `2>&1` is a frequently tested concept. Explain it as:
> "Send file descriptor 2 (stderr) to the same place as file descriptor 1 (stdout)."
> The `&` before `1` tells the shell "this is a file descriptor, not a file named `1`."

## Input Redirection

### `<` — Redirect stdin

```bash
# Feed a file as input to a command (instead of keyboard)
sort < words.txt

# Count lines from a file via stdin
wc -l < employees.csv
# (note: this won't print the filename, unlike wc -l employees.csv)

# Use with grep
grep "Engineering" < employees.csv

# Mail the contents of a file (if mail is configured)
# mail user@example.com < message.txt

# Combine input and output redirection
sort < words.txt > sorted_words.txt
# Read from words.txt, sort it, write to sorted_words.txt
```

### `<<` — Here Document (Heredoc)

A here document feeds multiple lines of inline text as stdin to a command.

```bash
# Basic heredoc — create a file with multi-line content
cat << EOF > server_config.txt
hostname=webserver01
ip_address=192.168.1.100
port=8080
environment=production
EOF

# Use with any command that reads stdin
sort << END
banana
apple
cherry
date
END
# Output:
# apple
# banana
# cherry
# date

# Heredoc with variable expansion (default behavior)
NAME="Student"
cat << EOF
Hello, $NAME!
Today is $(date).
Your home is $HOME.
EOF

# Heredoc WITHOUT variable expansion (quote the delimiter)
cat << 'EOF'
This will print literally: $NAME
No expansion: $(date)
Dollar sign: $HOME
EOF

# Use heredoc to create a script
cat << 'SCRIPT' > backup.sh
#!/bin/bash
echo "Starting backup..."
tar -czf backup.tar.gz /home/student/documents
echo "Backup complete!"
SCRIPT
chmod +x backup.sh

# Feed commands to an interactive program
mysql -u root << SQL
USE mydb;
SELECT * FROM users;
SQL
```

### `<>` — Open File for Reading AND Writing

```bash
# Open a file for both reading and writing on fd 0
# This is rarely used in practice but important to know
exec 3<> myfile.txt     # Open myfile.txt on fd 3 for read/write
read line <&3           # Read a line from fd 3
echo "New content" >&3  # Write to fd 3
exec 3>&-               # Close fd 3
```

> **Teaching Note:** `<>` is an advanced operator that students won't use often. Mention it for completeness but don't dwell on it. Focus on `<`, `<<`, `>`, `>>`, `2>`, and `&>`.

## Redirection Summary Table

| Operator  | Name                               | Description                                          |
|-----------|-------------------------------------|------------------------------------------------------|
| `>`       | Redirect stdout (overwrite)         | Send stdout to file, overwriting it                  |
| `>>`      | Redirect stdout (append)            | Send stdout to file, appending to it                 |
| `1>`      | Redirect stdout (explicit)          | Same as `>` — explicitly references fd 1             |
| `2>`      | Redirect stderr (overwrite)         | Send stderr to file, overwriting it                  |
| `2>>`     | Redirect stderr (append)            | Send stderr to file, appending to it                 |
| `&>`      | Redirect both (overwrite)           | Send stdout + stderr to file, overwriting            |
| `&>>`     | Redirect both (append)              | Send stdout + stderr to file, appending              |
| `2>&1`    | Merge stderr into stdout            | Send stderr to same destination as stdout            |
| `<`       | Redirect stdin                      | Feed file contents as input to command               |
| `<<`      | Here document                       | Feed inline multi-line text as input                 |
| `<>`      | Open for read/write                 | Open file for both input and output                  |

## The Pipe `|` — Connecting Commands

The pipe sends the **stdout** of one command as the **stdin** of the next command.

```
Command1  ──stdout──>  |  ──stdin──>  Command2  ──stdout──>  |  ──stdin──>  Command3
```

### Basic Pipe Usage

```bash
# Send output of ls to grep
ls -la | grep "\.txt$"

# Count files in a directory
ls -1 | wc -l

# Sort the output of a command
ps aux | sort -k 3 -rn | head

# Search command history
history | grep "apt"
```

## The `less` Command — Paginated Viewing

```bash
# View a file
less server.log

# View piped output
cat server.log | less
grep -r "error" /var/log/ | less
```

### Navigation in `less`

| Key              | Action                            |
|------------------|-----------------------------------|
| `Space` / `f`    | Page down                         |
| `b`              | Page up                           |
| `Down` / `j`     | Scroll down one line              |
| `Up` / `k`       | Scroll up one line                |
| `g`              | Go to beginning of file           |
| `G`              | Go to end of file                 |
| `/pattern`       | Search forward                    |
| `?pattern`       | Search backward                   |
| `n`              | Next search match                 |
| `N`              | Previous search match             |
| `q`              | Quit                              |
| `h`              | Help                              |

> **Teaching Note:** `less` is better than `more` in almost every way (ironic!). The key advantage: `less` allows **backward scrolling** and doesn't load the entire file into memory. Mnemonic: **"less is more."**

## The `more` Command — Older Paginated Viewing

```bash
# View a file
more server.log

# View piped output
dmesg | more
```

### Navigation in `more`

| Key              | Action                            |
|------------------|-----------------------------------|
| `Space`          | Page down                         |
| `Enter`          | Scroll down one line              |
| `b`              | Page up (may not work on all versions) |
| `/pattern`       | Search forward                    |
| `q`              | Quit                              |

> **Teaching Note:** `more` is older and less capable than `less`. Some minimal systems only have `more`. The main limitation: you can't scroll backwards (in most implementations). Always prefer `less` when available.

## The `tee` Command — Split Output

`tee` reads stdin and writes it to **both** stdout AND one or more files simultaneously.

```bash
# Write to a file AND display on screen
ls -la | tee directory_listing.txt

# Append instead of overwrite
echo "New log entry" | tee -a logfile.txt

# Write to multiple files at once
echo "Important message" | tee file1.txt file2.txt file3.txt

# Use in a pipeline (save intermediate results)
cat server.log | grep "ERROR" | tee errors.txt | wc -l
# Saves errors to file AND counts them
```

> **Teaching Note:** `tee` is named after a T-shaped pipe fitting — it splits the flow. Without `tee`, you'd have to choose between seeing output on screen OR saving it to a file. `tee` lets you do both.

## The `grep` Command with Pipes and Redirection

```bash
# Pipe command output to grep
ps aux | grep "apache2"
df -h | grep "/dev/sd"
cat /etc/passwd | grep "/bin/bash"

# Chain grep commands (AND logic — must match both)
cat server.log | grep "ERROR" | grep "database"

# Redirect grep results
grep "CRITICAL" server.log > critical_errors.txt

# Grep with redirection: separate matches and errors
grep "pattern" * 2> grep_errors.txt 1> grep_results.txt

# Grep with pipe to sort
grep "ERROR" server.log | sort

# Grep with pipe to sort by count (find most common errors)
grep "ERROR" server.log | sort | uniq -c | sort -rn
```

## Multiple Pipe Operations — Putting It All Together

This is where everything connects. Complex pipelines chain simple commands to perform powerful data processing.

### Pipeline Pattern: cat → grep → cut → sort → less

```bash
# Full analysis pipeline:
# 1. cat: read the file
# 2. grep: filter for Engineering department
# 3. cut: extract name and salary fields
# 4. sort: sort by salary
# 5. less: view results page-by-page
cat employees.csv | grep "Engineering" | cut -d ',' -f 1,4 | sort -t ',' -k 2 -rn | less
```

### Pipeline Pattern: cat → grep → tee → wc

```bash
# Find errors, save them, AND count them
cat server.log | grep "ERROR" | tee error_report.txt | wc -l
```

### Pipeline Pattern: cat → cut → sort → uniq

```bash
# Find unique departments and count employees in each
cat employees.csv | cut -d ',' -f 3 | sort | uniq -c | sort -rn
```

### Pipeline Pattern: grep → cut → sort → uniq → sort

```bash
# Find top 10 IP addresses in a log (most requests)
grep "GET\|POST" server.log | cut -d ' ' -f 1 | sort | uniq -c | sort -rn | head -10
```

### Pipeline Pattern: cat → grep → tee → cut → sort → less

```bash
# Complex: filter, save intermediate, extract, sort, view
cat employees.csv | grep -v "^Name" | tee all_data.txt | cut -d ',' -f 3,4 | sort -t ',' -k 1 | tee dept_salary.txt | less
```

### Pipeline with Redirection

```bash
# Pipeline output to file
cat employees.csv | grep "Engineering" | sort -t ',' -k 4 -rn > eng_by_salary.txt

# Pipeline with stderr handling
find / -name "*.log" 2>/dev/null | head -20

# Save pipeline output AND errors separately
cat employees.csv | grep "Marketing" 1> marketing.txt 2> pipe_errors.txt
```

### Real-World Pipeline Examples

```bash
# Who is logged in and what are they running?
w | tail -n +3 | sort

# Show disk usage sorted by size
du -sh /var/* 2>/dev/null | sort -rh | head -10

# Find the 10 most frequently used commands
history | awk '{print $2}' | sort | uniq -c | sort -rn | head -10

# Monitor a log file in real time, filtering for errors
tail -f /var/log/syslog | grep --line-buffered "error"

# Extract and sort unique email domains
cut -d '@' -f 2 emails.txt | sort -u

# Process CSV: get average salary by department (awk addition)
cat employees.csv | grep -v "^Name" | awk -F',' '{dept[$3]+=$4; count[$3]++} END {for (d in dept) print d, dept[d]/count[d]}' | sort -k 2 -rn
```

## `/dev/null` — The Black Hole

```bash
# Discard stdout
command > /dev/null

# Discard stderr
command 2> /dev/null

# Discard ALL output
command &> /dev/null

# Common use: check if command succeeds without seeing output
if grep -q "root" /etc/passwd > /dev/null 2>&1; then
    echo "root user exists"
fi
```

> **Teaching Note:** `/dev/null` is a special device file that discards anything written to it. Think of it as a trash can. It's extremely useful in scripts where you only care about the exit code, not the output.
