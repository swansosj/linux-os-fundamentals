How the Web Works

What actually happens behind the scenes when you visit a website?

Step 1: DNS Resolution

When you type http://www.lab.local into your browser, the first thing that happens is DNS resolution — translating the human-readable name into an IP address.

Browser: "What is the IP address of www.lab.local?" │ ▼ OS DNS Resolver (/etc/resolv.conf → nameserver 192.168.1.100) │ ▼ BIND9 DNS Server (192.168.1.100:53) │ │ Looks up zone file: /etc/bind/zones/db.lab.local │ Found: www IN A 192.168.1.100 │ ▼ Browser: "Got it! www.lab.local = 192.168.1.100"

You can see this yourself:

$ dig www.lab.local +short
192.168.1.100

$ nslookup www.lab.local
Server:		192.168.1.100
Address:	192.168.1.100#53
Name:	www.lab.local
Address: 192.168.1.100
Key File: /etc/resolv.conf — This file tells your computer which DNS server to ask. We pointed it to our lab server.

Step 2: TCP Connection

Now that the browser has the IP address, it opens a TCP connection to port 80 (HTTP) on the server. This is the famous "three-way handshake":

Browser (client) Nginx (server) │ │ │ ──── SYN ────────────────────────> │ 1. "I want to connect" │ │ │ <──── SYN-ACK ──────────────────── │ 2. "OK, I acknowledge" │ │ │ ──── ACK ────────────────────────> │ 3. "Great, we're connected" │ │ │ ═══ TCP CONNECTION ESTABLISHED ═══ │

You can verify the server is listening:

$ sudo ss -tlnp | grep :80
LISTEN  0  511  0.0.0.0:80  0.0.0.0:*  users:(("nginx",pid=1235,fd=6))

Step 3: HTTP Request

Over the TCP connection, the browser sends an HTTP request. Here's what it actually looks like:

GET /about.html HTTP/1.1
Host: www.lab.local
User-Agent: Mozilla/5.0 (X11; Linux x86_64)
Accept: text/html,application/xhtml+xml
Accept-Language: en-US,en;q=0.5
Accept-Encoding: gzip, deflate
Connection: keep-alive

The key parts:

PartExampleMeaning
MethodGETI want to retrieve a resource
Path/about.htmlWhich file I want
Host headerwww.lab.localWhich website (virtual host)
User-AgentMozilla/5.0...What browser I'm using

Try it yourself with curl:

$ curl -v http://www.lab.local/about.html 2>&1 | head -20

Step 4: Server Processing

Nginx receives the request and processes it:

  1. Checks server_name to match the right virtual host config
  2. Looks at the Nginx config file: /etc/nginx/sites-enabled/lab.local
  3. Finds root /var/www/lab.local;
  4. Reads the file: /var/www/lab.local/about.html
  5. Sends it back as an HTTP response
The symlink connection: Nginx looks in sites-enabled/ which contains a symbolic link to the real config in sites-available/. This is the ln -s command from lesson 6.1.1!

Step 5: HTTP Response

The server sends back the HTML content with an HTTP response:

HTTP/1.1 200 OK
Server: nginx/1.22.1
Date: Sun, 01 Mar 2026 15:00:00 GMT
Content-Type: text/html
Content-Length: 5432
Connection: keep-alive

<!DOCTYPE html>
<html>
  (... the HTML content of this page ...)
</html>
Status CodeMeaning
200 OKSuccess — here's your page
301 MovedPage has permanently moved
304 Not ModifiedYou already have the latest version (cached)
403 ForbiddenYou're not allowed to access this
404 Not FoundThis page doesn't exist
500 Server ErrorSomething broke on the server

Step 6: Browser Renders the Page

The browser receives the HTML and:

  1. Parses the HTML structure
  2. Finds the <link rel="stylesheet" href="css/style.css"> tag
  3. Sends another HTTP request for css/style.css
  4. Applies the CSS styles to the HTML elements
  5. Renders the final visual page you see
Every resource = another request! Each CSS file, image, JavaScript file, and font triggers a separate HTTP request to the server. A typical modern webpage makes 50-100 requests.

Watch the requests in real time:

$ sudo tail -f /var/log/nginx/lab.local.access.log
192.168.1.50 - - [01/Mar/2026:15:00:00] "GET /about.html HTTP/1.1" 200 5432
192.168.1.50 - - [01/Mar/2026:15:00:00] "GET /css/style.css HTTP/1.1" 200 4096
