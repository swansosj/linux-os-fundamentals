systemd Services

How systemctl manages the services powering this website.

Services Running This Website

Two systemd services work together to deliver this page to you:

nginx.service

Active

Role: Web server — serves HTML, CSS, and other files over HTTP (port 80)

Config: /etc/nginx/sites-enabled/lab.local

Docs Root: /var/www/lab.local/

Logs: /var/log/nginx/

$ sudo systemctl status nginx

bind9.service

Active

Role: DNS server — resolves www.lab.local to the server's IP address (port 53)

Config: /etc/bind/named.conf.local

Zones: /etc/bind/zones/

Logs: journalctl -u bind9

$ sudo systemctl status bind9

systemctl Command Reference

Checking Status

CommandWhat It Does
systemctl status nginxDetailed status with recent logs
systemctl is-active nginxQuick check: "active" or "inactive"
systemctl is-enabled nginxDoes it start on boot? "enabled" or "disabled"
systemctl is-failed nginxHas it crashed? "active" or "failed"

Controlling Services

CommandWhat It DoesAnalogy
sudo systemctl start nginxStart the service nowTurn the light ON
sudo systemctl stop nginxStop the service nowTurn the light OFF
sudo systemctl restart nginxStop then startFlip the switch off and on
sudo systemctl reload nginxRe-read config, no downtimeAdjust the dimmer without flipping

Boot Behavior

CommandWhat It DoesAnalogy
sudo systemctl enable nginxStart on every bootSet an alarm clock
sudo systemctl disable nginxDon't start on bootTurn off the alarm
sudo systemctl enable --now nginxEnable AND start right nowSet alarm AND wake up now
sudo systemctl mask nginxCompletely prevent startingThrow away the alarm clock
Remember: start/stop = right now (this session). enable/disable = on boot (survives reboot). You usually need both!

Try It: Stop This Website

Here's a live demo you can try (and then fix!):

# Step 1: Verify nginx is running
$ systemctl is-active nginx
active

# Step 2: Stop nginx
$ sudo systemctl stop nginx

# Step 3: Try to load the website → FAILS!
$ curl http://localhost
curl: (7) Failed to connect to localhost port 80: Connection refused

# Step 4: Check status — see it's dead
$ systemctl status nginx
● nginx.service - A high performance web server
     Active: inactive (dead)

# Step 5: Start it again
$ sudo systemctl start nginx

# Step 6: Website works again!
$ curl -I http://localhost
HTTP/1.1 200 OK

Viewing Logs with journalctl

journalctl reads the systemd journal — the centralized log for all services.

# Last 20 log lines for nginx
$ sudo journalctl -u nginx -n 20

# Follow nginx logs in real time
$ sudo journalctl -u nginx -f

# Show only errors
$ sudo journalctl -u nginx -p err

# Logs since last boot
$ sudo journalctl -u bind9 -b

# Logs from the last hour
$ sudo journalctl -u nginx --since "1 hour ago"

The Symlink Connection

Enabling a service in systemd and enabling an Nginx site both work the same way — symbolic links!

systemd "enable": systemctl enable nginx Creates: /etc/systemd/system/multi-user.target.wants/nginx.service → /lib/systemd/system/nginx.service Nginx "enable site": ln -s /etc/nginx/sites-available/lab.local /etc/nginx/sites-enabled/lab.local Creates: /etc/nginx/sites-enabled/lab.local → /etc/nginx/sites-available/lab.local Both use the same concept: Enabled = symlink exists (points to the real config) Disabled = symlink removed (config file stays safe)
# Verify the symlinks
$ ls -la /etc/nginx/sites-enabled/
lab.local -> /etc/nginx/sites-available/lab.local

$ ls -la /etc/systemd/system/multi-user.target.wants/ | grep nginx
nginx.service -> /lib/systemd/system/nginx.service

Listing Services

# All running services
$ systemctl list-units --type=service --state=running

# All enabled services (start on boot)
$ systemctl list-unit-files --type=service --state=enabled

# Failed services
$ systemctl list-units --type=service --state=failed

# View the actual unit file
$ systemctl cat nginx.service
