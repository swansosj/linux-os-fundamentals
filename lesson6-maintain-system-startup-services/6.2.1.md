# 6.2.1 — Managing Services with `systemctl`

---

# Part A: Introduction to `systemctl`

`systemctl` is the primary command for managing **systemd** services, targets, and units. It replaces the older `service` and `chkconfig`/`update-rc.d` commands.

---

## Core Concepts

### What is a Unit?

A **unit** is anything systemd can manage. The most common unit types:

| Unit Type    | Extension    | Purpose                                  | Example                     |
|------------- |------------- |------------------------------------------|-----------------------------|
| **Service**  | `.service`   | A daemon or process                      | `nginx.service`             |
| **Socket**   | `.socket`    | An IPC or network socket                 | `sshd.socket`               |
| **Target**   | `.target`    | A group of units (like a runlevel)       | `multi-user.target`         |
| **Timer**    | `.timer`     | A scheduled task (like cron)             | `apt-daily.timer`           |
| **Mount**    | `.mount`     | A filesystem mount point                 | `home.mount`                |
| **Path**     | `.path`      | Watches a filesystem path                | `cups.path`                 |

> **Teaching Note:** For this lesson, we focus on `.service` units. When you say "manage a service," you mean start, stop, enable, disable, and check the status of a `.service` unit.

### Where Unit Files Live

| Path                     | Purpose                                     |
|--------------------------|---------------------------------------------|
| `/lib/systemd/system/`   | Vendor/package-provided unit files (default) |
| `/etc/systemd/system/`   | Admin overrides and custom units (highest priority) |
| `/run/systemd/system/`   | Runtime-generated units (temporary)          |

Priority: `/etc/` > `/run/` > `/lib/`

---

## `systemctl` — Service Management Commands

### Checking Service Status

```bash
# Show detailed status of a service
systemctl status nginx
# Shows: loaded state, active state, PID, memory, recent logs

# Short check — is it active?
systemctl is-active nginx
# Output: active  (or: inactive)

# Is it enabled (starts on boot)?
systemctl is-enabled nginx
# Output: enabled  (or: disabled)

# Is it failed?
systemctl is-failed nginx
# Output: active  (or: failed)
```

**Key fields:**

|-------------|--------------------------------------------------------|

| `Tasks`     | Number of tasks/threads                                |
| `Memory`    | Current memory usage                                   |
| `CGroup`    | Control group tree showing child processes              |
| Log lines   | Most recent journal entries for this service            |

### Starting and Stopping Services

```bash
# Start a service NOW
sudo systemctl start nginx

# Stop a service NOW
sudo systemctl stop nginx

# Restart a service (stop + start)
sudo systemctl restart nginx

# Reload configuration WITHOUT restarting (zero downtime)
sudo systemctl reload nginx

# Reload if supported, otherwise restart
sudo systemctl reload-or-restart nginx
```

### Enabling and Disabling Services (Boot Behavior)

```bash
# Enable — start automatically on boot
sudo systemctl enable nginx

# Disable — do NOT start on boot
sudo systemctl disable nginx

# Enable AND start right now (two in one)
sudo systemctl enable --now nginx

# Disable AND stop right now
sudo systemctl disable --now nginx
```

> **Teaching Note:** The distinction between start/stop and enable/disable is crucial:
> - `start` / `stop` = **right now** (this session only)
> - `enable` / `disable` = **on boot** (persistent across reboots)
> - You often need BOTH: `enable` + `start` to make a service run now AND survive reboot.
> - The `--now` flag is a shortcut that does both at once.

### Masking Services (Hard Disable)

```bash
# Mask — completely prevent a service from starting (even manually)
sudo systemctl mask nginx
# Creates symlink: /etc/systemd/system/nginx.service -> /dev/null

# Try to start a masked service
sudo systemctl start nginx
# Error: Unit nginx.service is masked.

# Unmask — undo the mask
sudo systemctl unmask nginx
```

> **Teaching Note:** Masking is stronger than disabling. A disabled service can still be started manually. A masked service **cannot** be started at all until unmasked. Use it when you absolutely want to prevent a service from running.

## Listing and Inspecting Units

```bash
# List all loaded services
systemctl list-units --type=service

# List all services (including inactive)
systemctl list-units --type=service --all

# List only running services
systemctl list-units --type=service --state=running

# List only failed services
systemctl list-units --type=service --state=failed

# List all enabled services
systemctl list-unit-files --type=service --state=enabled

# List timers (scheduled tasks)
systemctl list-timers

# Show full unit file contents
systemctl cat nginx.service

# Show all properties of a service
systemctl show nginx.service

# Show a specific property
systemctl show nginx.service -p MainPID
systemctl show nginx.service -p ActiveState
```

---

## Viewing Logs with `journalctl`

`journalctl` reads the systemd journal — the centralized log for all services managed by systemd.

```bash
# Show all logs for a service
sudo journalctl -u nginx

# Show logs since last boot
sudo journalctl -u nginx -b

# Show last 50 lines
sudo journalctl -u nginx -n 50

# Follow logs in real time (like tail -f)
sudo journalctl -u nginx -f

# Show logs since a specific time
sudo journalctl -u nginx --since "2026-03-01 10:00:00"
sudo journalctl -u nginx --since "1 hour ago"

# Show only error-level and above
sudo journalctl -u nginx -p err

# Show logs for ALL units (recent 100 lines)
sudo journalctl -n 100

# Show logs for the current boot only
sudo journalctl -b

# Show disk usage of journal
sudo journalctl --disk-usage
```

---

## Reloading systemd After Config Changes

```bash
# After editing any unit file, tell systemd to re-read them
sudo systemctl daemon-reload

# Then restart the service to apply
sudo systemctl restart nginx
```

> **⚠️ Critical:** Always run `daemon-reload` after modifying a `.service` file. Without it, systemd uses the old cached version.

---

## Targets (Runlevels in systemd)

```bash
# Show current target (runlevel equivalent)
systemctl get-default
# Output: graphical.target (or multi-user.target for servers)

# Change default boot target
sudo systemctl set-default multi-user.target    # Text mode (no GUI)
sudo systemctl set-default graphical.target      # GUI mode

# Switch target right now (like changing runlevel live)
sudo systemctl isolate multi-user.target
sudo systemctl isolate rescue.target             # Single-user mode

# Reboot / shutdown via systemctl
sudo systemctl reboot
sudo systemctl poweroff
sudo systemctl halt
```

### Target ↔ Runlevel Mapping

| SysVinit Runlevel  | systemd Target            | Description          |
|--------------------|---------------------------|----------------------|
| 0                  | `poweroff.target`         | Halt / power off     |
| 1                  | `rescue.target`           | Single-user / rescue |
| 2, 3, 4            | `multi-user.target`       | Multi-user, text     |
| 5                  | `graphical.target`        | Multi-user + GUI     |
| 6                  | `reboot.target`           | Reboot               |

# Part B: Setting Up a DNS Server (BIND9)

## Overview

We'll install and configure **BIND9** (Berkeley Internet Name Domain) — the most widely used DNS server on Linux. This will let students point their machines to your sandbox server and resolve custom DNS records.

### What We're Building

```bash
 Student Machines                   Your Sandbox Server
┌──────────────┐                  ┌──────────────────────┐
│ DNS Client   │ ──── query ───>  │  BIND9 DNS Server    │
│              │                  │  ns1.lab.local       │
│ resolv.conf: │                  │  192.168.x.x         │
│ nameserver   │ <── response ──  │                      │
│ 192.168.x.x  │                  │  Also serves:        │
│              │                  │  www.lab.local →     │
└──────────────┘                  │    Nginx web server  │
                                  └──────────────────────┘
```

> **Before you begin:** Replace `192.168.1.100` throughout with your actual server IP. Use `ip addr show` or `hostname -I` to find it.

---

## Step 1: Install BIND9

```bash
sudo apt update
sudo apt install -y bind9 bind9utils bind9-doc dnsutils
```

### Verify installation

```bash
# Check that named (the BIND daemon) is installed
which named
# /usr/sbin/named

# Check version
named -v
# BIND 9.18.x (Extended Support Version)

# Check the service status
sudo systemctl status named
# OR (Debian uses 'named' or 'bind9' — check which)
sudo systemctl status bind9
```

---

## Step 2: Understand BIND9 File Layout

| Path                          | Purpose                          |
|-------------------------------|----------------------------------|
| `/etc/bind/named.conf`        | Main config (includes others)    |
| `/etc/bind/named.conf.options`| Server options (forwarders, etc.)|
| `/etc/bind/named.conf.local`  | Your zone declarations           |
| `/etc/bind/zones/`            | Zone data files (we'll create)   |

---

## Step 3: Configure BIND9 Options

Edit `/etc/bind/named.conf.options`:

```bash
sudo nano /etc/bind/named.conf.options
```

Replace contents with:

```
// /etc/bind/named.conf.options
// BIND9 Options — Lab DNS Server

options {
    directory "/var/cache/bind";

    // Allow queries from any network (adjust for your lab subnet)
    allow-query { any; };

    // Allow recursive queries (needed for full DNS resolution)
    recursion yes;
    allow-recursion { any; };

    // Forward unresolved queries to public DNS
    forwarders {
        8.8.8.8;        // Google DNS
        1.1.1.1;        // Cloudflare DNS
    };

    // Security: hide version info
    version "not available";

    // Listen on all interfaces
    listen-on { any; };
    listen-on-v6 { any; };

    // DNSSEC validation (auto uses built-in trust anchors)
    dnssec-validation auto;
};
```

---

## Step 4: Declare Your Zones

Edit `/etc/bind/named.conf.local`:

```bash
sudo nano /etc/bind/named.conf.local
```

Add the following:

```bash
// /etc/bind/named.conf.local
// Forward zone — lab.local
zone "lab.local" {
    type master;
    file "/etc/bind/zones/db.lab.local";
    allow-transfer { none; };
};

// Reverse zone — 192.168.1.x (adjust to your subnet)
// For 192.168.1.x, the reverse zone is 1.168.192.in-addr.arpa
zone "1.168.192.in-addr.arpa" {
    type master;
    file "/etc/bind/zones/db.192.168.1";
    allow-transfer { none; };
};
```

> **Teaching Note:** If your server's subnet is different (e.g., 10.0.0.x), change the reverse zone accordingly:

> - `10.0.0.x` → zone `"0.0.10.in-addr.arpa"`
> - `172.16.5.x` → zone `"5.16.172.in-addr.arpa"`

---

## Step 5: Create the Forward Zone File

```bash
sudo mkdir -p /etc/bind/zones
sudo nano /etc/bind/zones/db.lab.local
```

```
; /etc/bind/zones/db.lab.local
; Forward zone file for lab.local
;
$TTL    604800
@       IN      SOA     ns1.lab.local. admin.lab.local. (
                        2026030201      ; Serial (YYYYMMDD## — increment on changes)
                        3600            ; Refresh (1 hour)
                        1800            ; Retry (30 minutes)
                        604800          ; Expire (1 week)
                        86400 )         ; Negative Cache TTL (1 day)

; Name servers
@       IN      NS      ns1.lab.local.

; A records — name to IP mapping
ns1     IN      A       192.168.1.100
www     IN      A       192.168.1.100
web     IN      A       192.168.1.100
mail    IN      A       192.168.1.100
ftp     IN      A       192.168.1.100
db      IN      A       192.168.1.100
app     IN      A       192.168.1.100

; CNAME records — aliases
webserver   IN  CNAME   www.lab.local.
website     IN  CNAME   www.lab.local.
dns         IN  CNAME   ns1.lab.local.

; MX record — mail server
@       IN      MX  10  mail.lab.local.

; TXT record — example SPF record
@       IN      TXT     "v=spf1 mx -all"
```

---

## Step 6: Create the Reverse Zone File

```bash
sudo nano /etc/bind/zones/db.192.168.1
```

```
; /etc/bind/zones/db.192.168.1
; Reverse zone file for 192.168.1.x
;
$TTL    604800
@       IN      SOA     ns1.lab.local. admin.lab.local. (
                        2026030201      ; Serial
                        3600            ; Refresh
                        1800            ; Retry
                        604800          ; Expire
                        86400 )         ; Negative Cache TTL

; Name servers
@       IN      NS      ns1.lab.local.

; PTR records — IP to name mapping (last octet only)
100     IN      PTR     ns1.lab.local.
100     IN      PTR     www.lab.local.
```

## Step 7: Validate and Start

```bash
# Check the main config for syntax errors
sudo named-checkconf
# (no output = success)

# Check zone file syntax
sudo named-checkzone lab.local /etc/bind/zones/db.lab.local
# Output: zone lab.local/IN: loaded serial 2026030201
#         OK

sudo named-checkzone 1.168.192.in-addr.arpa /etc/bind/zones/db.192.168.1
# Output: OK

# Restart BIND9
sudo systemctl restart bind9

# Enable on boot
sudo systemctl enable bind9

# Verify it's running
sudo systemctl status bind9
```

## DNS Troubleshooting

```bash
# Check if BIND is listening on port 53
sudo ss -tlnp | grep :53
sudo ss -ulnp | grep :53
# Should see both TCP and UDP on port 53

# Check BIND logs
sudo journalctl -u bind9 -n 50
sudo journalctl -u bind9 -f         # Follow in real time

# Check for config errors
sudo named-checkconf
sudo named-checkzone lab.local /etc/bind/zones/db.lab.local

# Test with verbose dig output
dig @localhost www.lab.local +trace
dig @localhost www.lab.local +noall +answer +authority

# Check if firewall is blocking DNS
sudo iptables -L -n | grep 53
# If using ufw:
sudo ufw status
sudo ufw allow 53/tcp
sudo ufw allow 53/udp

# Dump the current zone data from memory
sudo rndc dumpdb -zones
sudo cat /var/cache/bind/named_dump.db | grep lab.local

# Reload zone without restarting
sudo rndc reload lab.local

# Flush BIND cache
sudo rndc flush
```

# Part C: Setting Up an Nginx Web Server

## Overview

**Nginx** (pronounced "engine-X") is the most popular web server as of 2025/2026, surpassing Apache in market share. It's known for high performance, low memory usage, and simplicity.

We'll build a multi-page demo website that shows students how web servers work under the hood.

---

## Step 1: Install Nginx

```bash
sudo apt update
sudo apt install -y nginx
```

### Verify installation

```bash
# Check the binary
which nginx
nginx -v
# Output: nginx version: nginx/1.22.x

# Check service status
sudo systemctl status nginx

# Enable and start
sudo systemctl enable --now nginx

# Verify it's listening on port 80
sudo ss -tlnp | grep :80
```

### Quick test — visit from a browser

```bash
# Get your server IP
hostname -I

# From any machine on the network, open:
# http://192.168.1.100
# You should see the default Nginx welcome page
```

---

## Step 2: Understand Nginx File Layout

| Path                              | Purpose                               |
|-----------------------------------|---------------------------------------|
| `/etc/nginx/nginx.conf`           | Main config file                      |
| `/etc/nginx/sites-available/`     | Available virtual host configs        |
| `/etc/nginx/sites-enabled/`       | Enabled virtual host configs (symlinks!) |
| `/var/www/html/`                  | Default document root                 |
| `/var/log/nginx/access.log`       | Access log                            |
| `/var/log/nginx/error.log`        | Error log                             |

> **Teaching Note:** Notice the `sites-available` → `sites-enabled` pattern uses **symbolic links** — directly ties back to lesson 6.1.1! Enabling a site means creating a symlink. Disabling means removing the symlink.

---

```
└── info.html           ← Server info (shows headers, etc.)
```

---

## Step 4: Configure the Nginx Virtual Host

```bash
sudo nano /etc/nginx/sites-available/lab.local
```

```nginx
# /etc/nginx/sites-available/lab.local
# Virtual host configuration for the lab website

server {
    listen 80;
    listen [::]:80;

    server_name lab.local www.lab.local web.lab.local;

    root /var/www/lab.local;
    index index.html;

    # Access and error logs
    access_log /var/log/nginx/lab.local.access.log;
    error_log  /var/log/nginx/lab.local.error.log;

    # Main location — try files, then directories, then 404
    location / {
        try_files $uri $uri/ =404;
    }

    # Custom 404 error page
    error_page 404 /404.html;
    location = /404.html {
        internal;
    }

    # Server info page — add custom headers for teaching
    location = /info.html {
        add_header X-Served-By "Nginx on Debian";
        add_header X-Lab "FSCJ Linux Fundamentals";
        try_files $uri =404;
    }

    # Deny access to hidden files
    location ~ /\. {
        deny all;
    }
}
```

### Enable the site

```bash
# Create symlink to enable (THIS is the ln -s lesson tie-in!)
sudo ln -s /etc/nginx/sites-available/lab.local /etc/nginx/sites-enabled/lab.local

# Show the symlink
ls -la /etc/nginx/sites-enabled/
# lab.local -> /etc/nginx/sites-available/lab.local

# Optionally remove the default site
sudo rm /etc/nginx/sites-enabled/default

# Test Nginx configuration syntax
sudo nginx -t
# Output: nginx: configuration file /etc/nginx/nginx.conf test is successful

# Reload Nginx to apply
sudo systemctl reload nginx
```

> **Teaching Note:** Point out to students that enabling an Nginx site is literally creating a symbolic link (`ln -s`) — exactly what we learned in 6.1.1. Disabling a site is removing that symlink (`rm`). The actual config file stays safe in `sites-available/`.

## Nginx Troubleshooting

```bash
# Test configuration syntax
sudo nginx -t

# Check if Nginx is running
sudo systemctl status nginx
sudo ss -tlnp | grep :80

# View access logs (who is hitting the server)
sudo tail -f /var/log/nginx/lab.local.access.log

# View error logs
sudo tail -f /var/log/nginx/lab.local.error.log

# View ALL Nginx errors
sudo tail -f /var/log/nginx/error.log

# View systemd journal logs
sudo journalctl -u nginx -f

# Check file permissions (common issue!)
ls -la /var/www/lab.local/
# Files should be owned by www-data (or at least world-readable)
namei -l /var/www/lab.local/index.html
# Shows permissions for every directory in the path

# Check which sites are enabled
ls -la /etc/nginx/sites-enabled/

# Check Nginx process info
ps aux | grep nginx
# Should see: master process (root) + worker processes (www-data)

# Reload vs Restart
sudo systemctl reload nginx     # Graceful — no dropped connections
sudo systemctl restart nginx    # Hard restart — brief downtime

# Check for port conflicts
sudo ss -tlnp | grep :80
# If Apache is also installed, it may conflict:
sudo systemctl stop apache2
sudo systemctl disable apache2
```
