# 6.1.1 — Finding the System Initialization Method

---

## Overview

Every Linux system uses an **init system** — the very first process (PID 1) that the kernel starts after booting. It is responsible for:

- Starting all other processes and services
- Managing system runlevels or targets
- Handling service dependencies
- Reaping orphaned (zombie) processes

The two main init systems you'll encounter:

| Init System   | Used By                                    | Config Location             | Command        |
|---------------|--------------------------------------------|-----------------------------|----------------|
| **systemd**   | Debian 8+, Ubuntu 15.04+, RHEL 7+, Fedora | `/etc/systemd/`, `/lib/systemd/` | `systemctl`    |
| **SysVinit**  | Older Debian, older Ubuntu, legacy systems | `/etc/init.d/`, `/etc/rc*.d/`    | `service`, `update-rc.d` |

> **Teaching Note:** Virtually all modern Debian/Ubuntu systems use **systemd**. However, students need to understand SysVinit because exam objectives include identifying which init system is in use, and some legacy/embedded systems still use it.

---

## How to Identify Your Init System

### Method 1: Check PID 1 directly

```bash
# The init system is ALWAYS PID 1. Check what it is:
ps -p 1 -o comm=
# Output on systemd system: systemd
# Output on SysVinit system: init
```

### Method 2: Use `which` + `readlink` to follow `/sbin/init`

```bash
# /sbin/init is ALWAYS present — it's the traditional path to the init binary.
# On systemd systems, it's a symlink. Let's trace it.

# Step 1: Where is init?
which init
# Output: /usr/sbin/init  (or /sbin/init)

# Step 2: Is it a real file or a symlink?
ls -la /sbin/init
# Output (systemd):  /sbin/init -> /lib/systemd/systemd
# Output (SysVinit): /sbin/init  (regular file, no arrow)

# Step 3: Follow the symlink to the real binary
readlink /sbin/init
# Output (systemd): /lib/systemd/systemd
# Output (SysVinit): (nothing — it's not a symlink)
```

### Method 3: Check for systemd-specific paths

```bash
# If this directory exists, the system uses systemd
ls /run/systemd/system
# Exists → systemd
# "No such file" → NOT systemd (likely SysVinit or Upstart)

# Check the systemd version
systemctl --version
# Output: systemd 252 (252.22-1~deb12u1)
```

### Method 4: Check with `stat` and `file`

```bash
# What type of file is /sbin/init?
file /sbin/init
# Output: /sbin/init: symbolic link to /lib/systemd/systemd

stat /sbin/init
```

> **Teaching Note:** The key takeaway for students: **"/sbin/init is PID 1, and on modern systems it's a symbolic link pointing to systemd."** Understanding this symlink chain is why we're covering `ln`, `readlink`, and `which` together — they're the tools for tracing this path.

# The `which` Command

`which` locates the executable file associated with a command by searching the directories in your `$PATH`.

## Basic Usage

```bash
# Find the full path of a command
which init
# Output: /usr/sbin/init

which bash
# Output: /usr/bin/bash

which systemctl
# Output: /usr/bin/systemctl

which python3
# Output: /usr/bin/python3
```

## The `-a` Option — Show ALL Matches

```bash
# Show all matching executables found in PATH (not just the first)
which -a python
# Output:
# /usr/bin/python
# /usr/local/bin/python

which -a python3
# If multiple Python versions are installed, shows all paths
```

## How `which` Works — Understanding $PATH

```bash
# which searches these directories IN ORDER:
echo $PATH
# Output: /usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

# The FIRST match wins — this is why order matters
# If /usr/local/bin/python exists AND /usr/bin/python exists,
# which python returns whichever is first in $PATH
```

## Using `which` to Verify Installation

```bash
# Check if a program is installed and in PATH
which docker
# If installed: /usr/bin/docker
# If NOT installed: (no output, exit code 1)

# Use in a conditional
which nginx && echo "nginx is installed" || echo "nginx is NOT found in PATH"

# Check multiple commands at once
which ls cat grep find sort
```

## `which` vs Commands That Look Similar

```bash
# which — searches $PATH only, finds external executables
which echo
# Output: /usr/bin/echo

# type — shell builtin, knows about aliases, builtins, functions too
type echo
# Output: echo is a shell builtin

# whereis — searches standard directories, finds binary + man + source
whereis echo
# Output: echo: /usr/bin/echo /usr/share/man/man1/echo.1.gz
```

> **Teaching Note:** `which` is limited — it ONLY finds executable files in `$PATH`. It does NOT find:
> - Shell builtins (`cd`, `echo`, `alias`)
> - Shell functions
> - Aliases
>
> For this lesson, `which` matters because we use it to find `/sbin/init` — the entry point into tracing the init system.

# The `ln` Command — Creating Links

`ln` creates links between files. Linux has two types of links, and understanding them is essential for understanding how `/sbin/init` points to systemd.

## Types of Links

### Symbolic Links (Soft Links) — `ln -s`

A symbolic link is a **pointer file** that contains the path to another file. Think of it like a **shortcut** in Windows.

```
symlink_file  →  points to  →  target_file
```

- Has its own inode (it's a separate file)
- Can point to files OR directories
- Can cross filesystem boundaries
- **Breaks** if the target is moved or deleted (dangling link)
- Shows as `l` in `ls -l` output

### Hard Links — `ln` (no `-s`)

A hard link is an **additional name** for the same file data on disk. Both names point to the same inode.

```
name_A  ──┐
           ├──>  [same inode / same data on disk]
name_B  ──┘
```

- Shares the same inode as the original
- Cannot link to directories (on most systems)
- Cannot cross filesystem boundaries
- File data persists until ALL hard links are removed
- Indistinguishable from the "original" — there is no original

## Creating Symbolic Links

### Basic Syntax

```bash
ln -s TARGET LINK_NAME
#     ^^^^^^ ^^^^^^^^
#     │      └── The new symlink to create
#     └── The existing file/dir to point to
```

### Examples

```bash
# Create a symlink to a file
echo "Hello from the original" > /tmp/original.txt
ln -s /tmp/original.txt ~/shortcut.txt

# Verify
ls -l ~/shortcut.txt
# Output: lrwxrwxrwx 1 student student 20 Mar 2 10:00 shortcut.txt -> /tmp/original.txt

# Read through the symlink (follows it automatically)
cat ~/shortcut.txt
# Output: Hello from the original

# Create a symlink to a directory
ln -s /var/log ~/logs
ls ~/logs         # Shows contents of /var/log

# Create a symlink with a relative path
cd /tmp
echo "test" > myfile.txt
ln -s myfile.txt mylink.txt
ls -l mylink.txt
# Output: mylink.txt -> myfile.txt
```

### The `-f` Option — Force (Overwrite Existing)

```bash
# If the link already exists, -f replaces it
ln -sf /new/target existing_link

# Useful when updating where a symlink points
ln -sf /usr/bin/python3.11 /usr/local/bin/python3
```

### The `-v` Option — Verbose

```bash
ln -sv /var/log ~/logs
# Output: '/home/student/logs' -> '/var/log'
```

### Symlinks in System Initialization Context

```bash
# This is the KEY example for this lesson:
ls -la /sbin/init
# Output: /sbin/init -> /lib/systemd/systemd

# The system uses a symlink so that /sbin/init (the traditional path)
# can point to whatever init system is installed.
# On systemd systems: /sbin/init → /lib/systemd/systemd
# On SysVinit systems: /sbin/init is a regular binary (not a symlink)

# Other important system symlinks:
ls -la /usr/sbin/init
ls -la /bin
# On modern Debian: /bin -> usr/bin  (merged /usr)
ls -la /sbin
# On modern Debian: /sbin -> usr/sbin
```

### Broken (Dangling) Symlinks

```bash
# Create a symlink, then delete the target
echo "I exist" > /tmp/target.txt
ln -s /tmp/target.txt ~/mylink.txt
cat ~/mylink.txt        # Works

rm /tmp/target.txt
cat ~/mylink.txt        # ERROR: No such file or directory
ls -l ~/mylink.txt      # Still shows the link (often in red)

# Find broken symlinks
find /home -xtype l
```

> **Teaching Note:** Demonstrate the broken symlink — students need to see that a symlink can exist even when its target doesn't. This is a common source of confusing errors on Linux systems. The symlink itself occupies disk space (just the path string), but it's useless if the target is gone.

## Creating Hard Links

### Basic Syntax

```bash
ln TARGET LINK_NAME      # No -s flag
```

### Examples

```bash
# Create a hard link
echo "Shared data" > /tmp/file_a.txt
ln /tmp/file_a.txt /tmp/file_b.txt

# Both names point to the SAME inode
ls -li /tmp/file_a.txt /tmp/file_b.txt
# Output:
# 1234567 -rw-r--r-- 2 student student 12 Mar 2 10:00 file_a.txt
# 1234567 -rw-r--r-- 2 student student 12 Mar 2 10:00 file_b.txt
#  ^^^^^^                ^
#  same inode            link count is 2

# Modify through one name — both see the change
echo "Modified!" >> /tmp/file_a.txt
cat /tmp/file_b.txt
# Output:
# Shared data
# Modified!

# Delete one name — the other still works (data not lost)
rm /tmp/file_a.txt
cat /tmp/file_b.txt    # Still works! Data persists until link count reaches 0.
```

### Hard Links vs Symlinks

```bash
# HARD LINK: deleting "original" does NOT affect the link
echo "test data" > /tmp/hard_orig.txt
ln /tmp/hard_orig.txt /tmp/hard_link.txt
rm /tmp/hard_orig.txt
cat /tmp/hard_link.txt     # ✓ Still works

# SOFT LINK: deleting target BREAKS the link
echo "test data" > /tmp/soft_orig.txt
ln -s /tmp/soft_orig.txt /tmp/soft_link.txt
rm /tmp/soft_orig.txt
cat /tmp/soft_link.txt     # ✗ Error: No such file or directory
```

## Comparison Table: Hard Links vs Symbolic Links

| Feature                  | Hard Link              | Symbolic Link (Soft Link)  |
|--------------------------|------------------------|----------------------------|
| Command                  | `ln target link`       | `ln -s target link`        |
| Has its own inode?       | No (shares inode)      | Yes (separate inode)       |
| Can link to directories? | No                     | Yes                        |
| Cross filesystem?        | No                     | Yes                        |
| Target deleted?          | Data survives          | Link breaks (dangling)     |
| `ls -l` identifier       | Regular file, link count > 1 | `l` at start, arrow `->` |
| `ls -i` (inode)          | Same inode as target   | Different inode            |
| Size                     | Same as target         | Size of the path string    |
| Use case                 | Backup, data safety    | Shortcuts, version switching |

> **Teaching Note:** Use this analogy:
> - **Hard link** = Two names in a phone book for the same person. Delete one entry, the person still exists.
> - **Symbolic link** = A note that says "Go ask Bob." Delete Bob, the note is useless.
>
> For system init, the key concept is that `/sbin/init` is a **symbolic link** to the actual init binary. This lets the system swap init systems by changing where the symlink points.

# The `readlink` Command — Resolve Symbolic Links

`readlink` prints the target of a symbolic link. It answers: **"Where does this symlink point?"**

## Basic Usage

```bash
# Show what a symlink points to (one level)
readlink /sbin/init
# Output: /lib/systemd/systemd

readlink /usr/bin/python3
# Output: python3.11  (or whatever version is linked)

# If the file is NOT a symlink, readlink prints nothing
readlink /usr/bin/bash
# (no output — bash is a regular file, not a symlink)
```

## The `-f` Option — Follow the ENTIRE Chain (Canonical Path)

This is the most important option. `-f` resolves ALL symlinks in the entire path and returns the **final, absolute, real path**.

```bash
# Follow the complete chain to the real file
readlink -f /sbin/init
# Output: /usr/lib/systemd/systemd
# (resolves /sbin -> /usr/sbin AND the init -> systemd symlink)

readlink -f /usr/bin/python3
# Output: /usr/bin/python3.11

readlink -f /bin/bash
# Output: /usr/bin/bash
# (resolves /bin -> /usr/bin on merged-usr systems)

# Even works on regular files (resolves the directory path)
readlink -f /etc/hostname
# Output: /etc/hostname (already canonical)
```

## The `-e` Option — Like `-f` but All Components Must Exist

```bash
# -e requires every component of the path to exist
readlink -e /sbin/init
# Output: /usr/lib/systemd/systemd  (if it exists)

readlink -e /sbin/nonexistent
# (no output — the target doesn't exist)

# -f would still try to resolve what it can:
readlink -f /sbin/nonexistent
# Output: /usr/sbin/nonexistent  (resolves /sbin even though file doesn't exist)
```

## The `-m` Option — Don't Require Anything to Exist

```bash
# -m resolves what it can, doesn't care about existence
readlink -m /sbin/fake/path/file.txt
# Output: /usr/sbin/fake/path/file.txt  (resolved /sbin -> /usr/sbin)
```

## Tracing the Init System — Complete Walkthrough

This is the core demo for this lesson. Walk through this step by step:

```bash
# Step 1: What is PID 1?
ps -p 1 -o comm=
# Output: systemd

# Step 2: Find the init binary in PATH
which init
# Output: /usr/sbin/init

# Step 3: Is it a symlink?
ls -la /usr/sbin/init
# Output: lrwxrwxrwx 1 root root 20 ... /usr/sbin/init -> /lib/systemd/systemd

file /usr/sbin/init
# Output: /usr/sbin/init: symbolic link to /lib/systemd/systemd

# Step 4: Where does it point? (one level)
readlink /usr/sbin/init
# Output: /lib/systemd/systemd

# Step 5: Resolve the FULL canonical path
readlink -f /usr/sbin/init
# Output: /usr/lib/systemd/systemd

# Step 6: What about /sbin/init? (traditional path)
ls -la /sbin
# On merged-usr systems: /sbin -> usr/sbin

readlink -f /sbin/init
# Output: /usr/lib/systemd/systemd  (same destination!)

# Step 7: Confirm the real binary exists and is executable
ls -la "$(readlink -f /sbin/init)"
file "$(readlink -f /sbin/init)"
# Output: ELF 64-bit LSB pie executable...
```

> **Diagram: The Symlink Chain on a Modern Debian System**
> ```
> /sbin/init
>     │  (symlink: /sbin -> usr/sbin)
>     ▼
> /usr/sbin/init
>     │  (symlink: init -> /lib/systemd/systemd)
>     ▼
> /lib/systemd/systemd
>     │  (symlink: /lib -> usr/lib)
>     ▼
> /usr/lib/systemd/systemd    ← THE REAL BINARY (PID 1)
> ```

## Other Useful `readlink` Examples

```bash
# Find the real path of Python
readlink -f $(which python3)
# Output: /usr/bin/python3.11

# Find where java really lives
readlink -f $(which java)
# Output: /usr/lib/jvm/java-17-openjdk-amd64/bin/java

# Resolve "." and ".." in paths
readlink -f /var/log/../log/./syslog
# Output: /var/log/syslog

# Find the canonical path of your current directory
readlink -f .

# Useful in scripts: get the real location of the running script
SCRIPT_DIR=$(readlink -f "$(dirname "$0")")
```

> **Teaching Note:** The key takeaway: `readlink -f` gives you the **truth** — the actual, no-symlinks, absolute path to a file. This is essential for system administration because so many system paths involve symlinks (especially on modern merged-usr Debian systems where `/bin`, `/sbin`, and `/lib` are all symlinks into `/usr/`).

# Introduction to the `ps` Command — Process Status

`ps` displays information about currently running processes. This is the starting point for understanding what's running on a system — including PID 1 (init).

## Basic Usage

```bash
# Show processes for the current user in the current terminal
ps
#   PID TTY          TIME CMD
#  1234 pts/0    00:00:00 bash
#  1256 pts/0    00:00:00 ps
```

## Common Option Styles

`ps` supports **three** different option styles (a historical artifact):

| Style    | Example    | Origin        |
|----------|------------|---------------|
| UNIX     | `ps -ef`   | System V      |
| BSD      | `ps aux`   | Berkeley      |
| GNU long | `ps --pid` | GNU extension |

> **Teaching Note:** Students will see both `ps -ef` and `ps aux` in the real world. They produce similar output with slightly different formatting. Both are correct — it's a matter of preference. For exams, know both.

## UNIX Style: `ps -ef`

```bash
# Show ALL processes, full format
ps -ef

# Output columns:
# UID   PID  PPID  C STIME TTY      TIME     CMD
# root    1     0  0 08:00 ?    00:00:05 /sbin/init
# root    2     0  0 08:00 ?    00:00:00 [kthreadd]
# ...

# Column meanings:
# UID   — User who owns the process
# PID   — Process ID
# PPID  — Parent Process ID
# C     — CPU utilization
# STIME — Start time
# TTY   — Terminal (? = no terminal / daemon)
# TIME  — Cumulative CPU time
# CMD   — The command that started the process
```

## BSD Style: `ps aux`

```bash
# Show ALL processes with detailed info
ps aux

# Output columns:
# USER   PID %CPU %MEM    VSZ   RSS TTY  STAT START   TIME COMMAND
# root     1  0.0  0.1 169536 13312 ?    Ss   08:00   0:05 /sbin/init
# root     2  0.0  0.0      0     0 ?    S    08:00   0:00 [kthreadd]
# ...

# Column meanings:
# USER    — User who owns the process
# PID     — Process ID
# %CPU    — CPU usage percentage
# %MEM    — Memory usage percentage
# VSZ     — Virtual memory size (KB)
# RSS     — Resident Set Size — actual physical memory used (KB)
# TTY     — Terminal (? = no terminal)
# STAT    — Process state (see below)
# START   — Start time
# TIME    — Cumulative CPU time
# COMMAND — Full command with arguments
```

### Process States (STAT column)

| Code | State           | Description                                    |
|------|-----------------|------------------------------------------------|
| `R`  | Running         | Currently executing or in the run queue         |
| `S`  | Sleeping        | Waiting for an event (interruptible)            |
| `D`  | Disk Sleep      | Waiting for I/O (uninterruptible)               |
| `T`  | Stopped         | Stopped by a signal (e.g., Ctrl+Z)             |
| `Z`  | Zombie          | Finished but parent hasn't read exit status     |
| `I`  | Idle            | Idle kernel thread                              |

**Additional characters:**

| Modifier | Meaning                          |
|----------|----------------------------------|
| `s`      | Session leader                   |
| `l`      | Multi-threaded                   |
| `+`      | In the foreground process group  |
| `<`      | High priority (not nice)         |
| `N`      | Low priority (nice)              |

```bash
# Examples of STAT:
# Ss  = Sleeping, session leader (common for daemons)
# R+  = Running, foreground
# Sl  = Sleeping, multi-threaded
# Z   = Zombie (needs investigation)
```

## Targeting Specific Processes

```bash
# Show only processes for a specific user
ps -u root
ps -u student

# Show a specific process by PID
ps -p 1
ps -p 1 -f              # Full format for PID 1

# Show only the command name of PID 1
ps -p 1 -o comm=
# Output: systemd

# Show custom columns
ps -eo pid,ppid,user,comm
ps -eo pid,%cpu,%mem,comm --sort=-%mem | head -15
# Top 15 processes by memory usage
```

## `ps` with Pipes and `grep`

```bash
# Find a specific process by name
ps aux | grep "apache2"
ps aux | grep "sshd"
ps aux | grep "systemd"

# Avoid matching the grep command itself
ps aux | grep "[a]pache2"
# The [a] trick: [a]pache2 matches "apache2" but not "[a]pache2" (the grep itself)

# Count processes
ps aux | wc -l

# Find all processes by a user
ps -ef | grep "^student"
```

## Using `ps` to Investigate the Init System

```bash
# What is PID 1?
ps -p 1 -f
# UID   PID  PPID  C STIME TTY      TIME     CMD
# root    1     0  0 08:00 ?    00:00:05 /sbin/init

ps -p 1 -o comm=
# systemd

ps -p 1 -o pid,ppid,comm,args
# PID  PPID COMM    ARGS
#   1     0 systemd /sbin/init

# Show all systemd-related processes
ps aux | grep "systemd"

# Show the process tree starting from PID 1
ps -ejH | head -30
# The -j shows job info, -H shows hierarchy (indented tree)

# Show process tree (forest view)
ps auxf | head -40
# f = forest (tree format with ASCII art)

# Or use pstree for a cleaner view
pstree -p | head -30
# Shows tree with PIDs

pstree -p 1 | head -20
# Tree starting from PID 1 (init/systemd)
```

## Custom Output Formatting

```bash
# Select exactly which columns to display
ps -eo pid,ppid,user,%cpu,%mem,stat,start,time,comm

# Sort by CPU usage (descending)
ps -eo pid,user,%cpu,%mem,comm --sort=-%cpu | head -10

# Sort by memory usage (descending)
ps -eo pid,user,%cpu,%mem,comm --sort=-%mem | head -10

# Sort by start time
ps -eo pid,user,lstart,comm --sort=start_time | tail -10

# Show processes with their full command line
ps -eo pid,user,args | grep -v "\[.*\]" | head -20
# Filters out kernel threads (shown in [brackets])
```

## Useful `ps` One-Liners

```bash
# Find zombie processes
ps aux | awk '$8 ~ /Z/ {print}'

# Find the parent of a zombie process
ps -eo pid,ppid,stat,comm | grep "Z"
# Then check the PPID to find who should be reaping it

# Count processes per user
ps -eo user | sort | uniq -c | sort -rn | head

# Show processes listening on network ports (combine with ss)
ps -eo pid,comm | grep -f <(ss -tlnp | grep -oP 'pid=\K[0-9]+')

# Show the top 5 memory consumers
ps -eo pid,user,%mem,rss,comm --sort=-%mem | head -6

# Find how long a process has been running
ps -p 1 -o pid,etime,comm
# ELAPSED shows days-hours:minutes:seconds
```

# Classroom Demo: Tracing the Init System

## Full Demo Script

Walk students through this entire sequence to tie all the commands together:

```bash
echo "============================================"
echo " Demo: Finding the System Initialization Method"
echo "============================================"

echo ""
echo "--- Step 1: What is the FIRST process (PID 1)? ---"
ps -p 1 -o pid,comm,args
echo ""

echo "--- Step 2: Where is the 'init' command? ---"

echo ""

echo "--- Step 3: What TYPE of file is it? ---"

echo ""

echo "--- Step 4: Where does the symlink point? (one level) ---"

echo ""

echo "--- Step 5: What is the FULL real path? ---"
readlink -f $(which init)

```

## Exercise 2: Working with Symbolic Links

```bash
# Create a project structure with symlinks
mkdir -p ~/lab6/{bin,lib,config}
echo '#!/bin/bash' > ~/lab6/bin/myapp_v1.sh
echo 'echo "App Version 1"' >> ~/lab6/bin/myapp_v1.sh
chmod +x ~/lab6/bin/myapp_v1.sh

echo '#!/bin/bash' > ~/lab6/bin/myapp_v2.sh
echo 'echo "App Version 2"' >> ~/lab6/bin/myapp_v2.sh
chmod +x ~/lab6/bin/myapp_v2.sh

# Create a "current version" symlink
ln -s ~/lab6/bin/myapp_v1.sh ~/lab6/bin/myapp
ls -l ~/lab6/bin/myapp
~/lab6/bin/myapp                # Runs v1

# "Upgrade" by changing the symlink
ln -sf ~/lab6/bin/myapp_v2.sh ~/lab6/bin/myapp
ls -l ~/lab6/bin/myapp
~/lab6/bin/myapp                # Now runs v2

readlink ~/lab6/bin/myapp
readlink -f ~/lab6/bin/myapp
```

**Questions for students:**

## Exercise 4: Process Investigation

```bash
# Find PID 1 information
ps -p 1 -f

# Show the process tree
pstree -p | head -20

# Find all processes owned by root
ps -u root -o pid,comm | head -20

# Find the top 5 CPU consumers
ps -eo pid,user,%cpu,comm --sort=-%cpu | head -6

# Find the top 5 memory consumers
ps -eo pid,user,%mem,rss,comm --sort=-%mem | head -6

# Count total running processes
ps aux | wc -l

# Find any zombie processes
ps aux | awk '$8 ~ /Z/'
```
